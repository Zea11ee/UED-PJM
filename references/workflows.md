# 典型流程清单

## 1) 外部需求 → 项目管理执行
1. 在「UED 需求总表」定位需求记录（表单/Inbox）。
2. 先查项目知识库（`project-kb/<PROJECT>.md`）与案例模板（如有），优先复用既有拆分策略。
3. 同步到「UED 项目管理 → 任务源」（字段映射见 sync-mapping）。
4. 补齐必填项：优先级、UED 负责人、归属项目。
5. 基于历史相似任务拆分 WBS 子任务（仅在知识库不足时扩大检索）。
6. 排期校验：北京时间、日期范围、预计工时、周末/假期。
7. 写入 WBS。
8. 落库后沉淀：更新 `feedback.md`，并默认执行项目KB追加脚本：  
   `python3 scripts/project-kb/append_project_kb_entry.py --project "<PROJECT>" --demand-id "<需求编号/源编号>" --module "<页面/模块>" --change "<新增/调整功能点>" --status "已排期" --parent-record-id "<parent_rec_id>" --wbs-record-ids "<wbs_rec_a,wbs_rec_b>"`。

## 2) 已有任务源 → WBS 更新
1. 查任务源记录，确认标题/优先级/负责人。
2. 查相似 WBS 结构（同项目/同类型）。
3. 规划任务与排期，征求确认（WBS 使用终端表格展示，禁止纯段落罗列）。
4. 写入/更新 WBS 子任务。
5. 记录反馈，并通过 `scripts/project-kb/append_project_kb_entry.py` 自动追加项目知识库事件条目（默认开启，重复事件自动跳过）。

## 2.1) 线上问题 → 任务源/WBS 创建
1. 任务源标题不含问题编号（编号写入源编号或备注即可）。
2. 需求方填写为线上问题提交人（如表单提交人）。
3. 描述补充复现链接与现象说明。
4. **落表前必须先出“父任务+WBS”草稿，包含源编号、归属项目、负责人、工时、日期，得到确认后再写入。**
5. 源编号获取规则：优先使用需求表/需求总表中的“需求编号/自动编号/手动编号”字段值，原样填入任务源的“源编号”；不得留空或自造编号。
6. 父任务创建时同步填写需求的期望上线日期/上线日期到任务源对应日期字段。

## 2.2) WBS 展示格式（新增）
1. 默认使用“终端表格样式”展示草案，建议放在代码块中以便对齐阅读。
2. WBS 列至少包含：序号、任务名称、负责人、状态、预计工时、启动日期、结束日期。
3. 父任务信息可单独用简表展示，便于用户一次确认后落库。

## 2.3) 节点倒排与轻重拆分（新增）
1. 先判定类型：轻量配置/联调、活动双阶段（Teasing/Launch）、问题修复。
2. 先锁里程碑日期（上线/开售），再反推前置任务日期。
3. 关键依赖默认规则：
   - `PRD` 必须早于开发；
   - 设计联调可与开发并行；
   - 上线走查必须在上线当天。
4. 轻量任务（<=2h）优先最小化 WBS，避免过度拆分。
5. 进入落库前，将“今天/明天”等相对时间统一转换为 Asia/Shanghai 绝对日期。

## 2.4) 落库稳定性控制（新增）
1. 先按源编号去重，存在则补子任务/更新，不重复建父任务。
2. 落库后回查父任务与WBS（按源编号、父任务关联）并返回 record_id。
3. 附件写入失败（跨 Base token）时，降级为描述记录来源链接/记录号，避免阻塞。

## 3) 需求大节点与节奏管理
1. 在「UED 项目大节点」定位项目。
2. 关联项目管理中的任务源/WBS 对齐。
3. 输出节奏摘要与风险点。

## 4) 人员排期/任务查询
1. 确认人员是否为 UED 部门。
2. UED 人员：优先查询「UED 项目管理 → WBS」按日期范围排期。
3. 非 UED 人员：若飞书任务接口不可用，明确说明限制与替代方案。
4. 新解析到的人员信息写入 `people-profiles.md`，供后续快速复用。

## 5) 项目管理方法/规范检索
1. 参考「UED 项目指导手册」统一规范与流程。
2. 若有新沉淀经验，更新指导手册与反馈记录。
